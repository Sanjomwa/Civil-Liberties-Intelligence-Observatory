# CLIO Implementation Roadmap — Engineering Reset

Status: Phase 3 deliverable, CLIO Engineering Reset. Evolutionary improvement, not a rewrite. Every item below references a specific finding in `architecture-assessment.md`, `technical-debt-inventory.md`, or `methodology-consistency-review.md` — nothing here is invented independently of those three documents. This roadmap is scoped to engineering health; the separate commercial/product roadmap from the Project Zero Review (`docs/01-product/roadmap.md`) is not repeated here and should be read alongside it, since a few items depend on each other across the two documents.

## Step 7 — Near real-time observatory capability (architecture and planning only, 2026-07-11)

Prompted by the grant-readiness and flagship-report planning in `docs/01-product/2026-07-11-execution-roadmap-grant-and-flagship.md`. No implementation is authorized by this step yet — it records the architectural decisions so implementation can start from a settled design rather than an open question.

**Finding: only OONI's data genuinely supports near-real-time refresh.** OONI probes submit measurements continuously; the only current gap is that nothing schedules the pipeline to consume that continuously — confirmed directly, not assumed: the 2026-07-06 cost audit found no GitHub workflow ever executes `bruin run`, so the OONI staging chain currently rebuilds 2–6 times per 90 days despite a declared `@daily` schedule. TD-58's clustering and partitioning work was explicitly preparing for this trigger ("conversion still deferred... trigger for revisiting: a real scheduled `bruin run` gets wired up") — this step is that trigger being pulled. ACLED, STOP, and Google Transparency Report are periodic by the nature of how each source itself produces data (ACLED's own weekly coding cadence; Google's semiannual reporting windows; STOP's export-based, non-continuous update pattern) — forcing any of them to refresh faster than their own source cadence would fabricate precision none of them actually have, repeating the exact grain-honesty mistake this project already caught and corrected at TD-40 and in ADR-0003's Contextual/Backdrop tier. "Near real-time observatory" means the OONI protocol layer updates continuously on its own dedicated surface, while every other source stays honestly labeled at its own real cadence.

**Architectural implications, decided now, built later:**
- A real, scheduled trigger for the OONI staging chain only (daily), using TD-58's already-shipped clustering/partitioning.
- Conversion of OONI staging to incremental/append materialization — the natural next step per the 2026-07-06 materialization-strategy audit, justified only once a real schedule exists (it does not yet).
- A dedicated near-real-time protocol-stress surface, separate from the composite-score/weekly-regime dashboard — this is a smaller change than it sounds, since ADR-0006 already established that OONI feeds no arithmetic term of `composite_pressure_score` at all. Making OONI real-time does not require touching the composite score's own weekly cadence.

**AI-layer implication:** the AI Architecture doctrine's Phase 6 (Country Intelligence Profiles) and Phase 7 (Election Monitoring Workflows) both assume some continuous or near-real-time capability as a prerequisite — this step is a genuine dependency of that later roadmap, not an independent track. Cost discipline (the doctrine's Principle 4) becomes load-bearing here specifically: the AI narration/interpretation layer should run on a coarser, state-transition-triggered cadence, not be re-invoked on every new raw OONI measurement — the Artifact Registry's "no repeated analytical recalculation if a valid artifact exists" prohibition is the mechanism that should enforce this, once Phase 1 of that doctrine is actually built (see Part D of the execution roadmap — it isn't yet).

**Sequencing:** (1) wire the scheduled OONI trigger; (2) convert to incremental materialization; (3) build the dedicated near-real-time surface; (4) wire AI narration with threshold-triggered, not continuous, regeneration. Do not revisit ACLED's, STOP's, or Google's cadence ahead of this — only if one of those sources' own update behavior genuinely changes first.

## What this roadmap deliberately does not include

No rewrite of the OONI path or ACLED path A's core logic — both are genuinely good and are the standard, not the problem. No new shared abstraction for data source integration (a base contract, a plugin system) until Google Transparency and Lumen have been brought up to a comparable methodological standard one at a time — there are not yet enough well-understood cases to generalize from safely. No multi-country parameterization before Step 4, since doing it earlier would multiply, not amortize, the hardcoding problem it is meant to fix.
