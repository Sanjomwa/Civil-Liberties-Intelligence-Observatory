# CLIO Methodology Consistency Review

Status: Phase 2 deliverable, CLIO Engineering Reset. This is an engineering audit of how the current codebase actually implements evidence, confidence, provenance, and interpretation — not a continuation of the archived Constitution's philosophical inquiry. Where this review touches the same ground the Constitution covered, it does so independently, from the code, and does not assume or restate the Constitution's conclusions. No rewrite is recommended here; the task is to understand and name what is conceptually consistent and what is not, and how serious each inconsistency is.

## 1. OONI Integration — the most conceptually coherent path

`stg.ooni_*` → `int.ooni_experiment_results` → `int.ooni_signals` → `features.protocol_daily_signals` → `intelligence.protocol_signal_regimes` / `protocol_lag_relationships` → `intelligence.protocol_relationships` → reporting marts.

This path cleanly separates observation from interpretation at every stage, assigns confidence through governed reference dimensions (`dim_censorship_confidence`, `dim_measurement_quality`, `dim_blocking_signals`) rather than inline literals in most places, and gates directional classification behind explicit statistical guardrails (sparse-window, zero-variance, low-sample flags) before any regime label is assigned. It is, on direct reading, the strongest single piece of methodological engineering in the codebase.

## 2. ACLED Integration — two paths of different methodological maturity

This is where the review's central finding sits. Two independent consumption paths exist for the same upstream data (`stg.acled_conflict_events`):

**Path A**, the sophisticated path: `int.acled_event_classification` assigns not one but two independent quantities — `classification_confidence` (how confident the classifier is in this specific label) and `methodology_risk_level` (how risky this event type is to classify consistently across countries at all) — a genuinely more sophisticated confidence model than OONI's single-axis scheme, because it separates "how sure are we" from "how comparable is this category across contexts" as two different questions. From there, `features.acled_pressure_signals` applies per-family baseline gating (protest/violence/suppression tracked independently), a calendar spine ensuring zero-event weeks are represented rather than silently absent, and explicit sparse/zero-variance guardrails, before `intelligence.acled_pressure_regimes` performs stateful, persistence-aware regime classification.

**Path B**, the legacy path: `stg.acled_conflict_events` → `marts.fact_country_pressure_daily` → `reporting.mart_political_stress_windows`, a direct aggregation with none of path A's confidence split, guardrails, or persistence logic.

**Confirmed 2026-07-04** (Architecture Assessment Section 6): the dashboard's live national pressure score is built from path B, not path A. This means the codebase's two most methodologically mature confidence architectures — OONI's dimension-table model and ACLED path A's dual confidence/risk split — do not both feed the number a client would actually see. Only OONI's does. This was the single most consequential open question in this review; it is now closed, and the remaining decision (what to do about path A) belongs to Step 3 of the Implementation Roadmap, not to this document.

## 3. Google Transparency and Lumen — the least methodologically developed paths

`int.google_pressure_periodized` and `int.lumen_pressure_daily` feed into `fact_takedown_activity` / `fact_country_pressure_daily` with no confidence or guardrail layer comparable to OONI or ACLED path A. For Google Transparency this is a maturity gap — a real data source integrated more simply than the others. For Lumen it is worse: the data entering `stg.lumen_requests` is entirely synthetic, generated by `scripts/lumen_parquet.py` (`np.random.seed(42)`, 5,000 fabricated rows), and nothing in the schema — no `is_synthetic` flag, no `data_source_authenticity` field — distinguishes it from real evidence anywhere downstream. This is not a methodology-sophistication gap like the others; it is a provenance-integrity gap, and it is the most serious finding in this review. A number derived from Lumen data looks, structurally, exactly like a number derived from real network measurement or real conflict events, and nothing in the pipeline prevents that number from reaching a report.

## 4. Evidence Modelling and Interpretation Pipeline — a real but non-uniform pattern

The recurring shape (observation → classified with confidence → statistically guarded feature → inferred regime) appears independently in the OONI path and in ACLED path A. This confirms, at the code level and independent of the Constitution's own analysis, that the pattern is real rather than imagined. It also adds one finding the Constitution's document-level review could not have made: the pattern is not applied uniformly even within a single source. ACLED path A has it; ACLED path B, built from the same upstream table, does not. Google Transparency and Lumen have it least of all. This means the pattern is a property of specific subsystems that were written carefully, not a property the architecture itself enforces. Nothing in the codebase — no base contract, no required schema, no Bruin asset template — requires a new data source integration to follow this shape. It happens when a contributor chooses to do it.

## 5. Data Provenance — strong on identifiers, unverified on lineage

Deterministic, content-hashed identifiers are used consistently (confirmed directly in `scripts/local_ingest_ooni.py`'s `stable_measurement_id()`, which falls back to a SHA-256 hash of a canonicalized identity dict when no natural ID exists). This is a mature, well-applied pattern. Separately, the `@bruin` `depends:` headers give a machine-readable lineage graph across the whole pipeline — but this lineage is declared, not verified. Nothing checks that a file's declared dependencies match what its SQL body actually selects from or joins against. The Architecture Assessment's central open question (Section 6, ACLED path A vs. reporting) is a direct symptom of this gap: if declared and actual dependencies had an automated cross-check, this question would already have a confirmed answer instead of a flagged one.

## 6. Severity Ranking of the Inconsistencies Found

1. **Critical — Lumen synthetic data with no provenance flag.** Must be fixed, or at minimum flagged and disclosed, before any client-facing report cites a Lumen-derived figure.
2. **Confirmed — only path B feeds the product.** No longer uncertain (verified 2026-07-04); this now blocks confident claims about the pressure score's evidentiary backing until Step 3 decides what to do about it, rather than blocking on verification itself.
3. **Medium-high — confidence-model maturity is inconsistent across sources.** OONI and ACLED path A have rich, well-designed confidence models; ACLED path B, Google Transparency, and Lumen do not. This is an evolutionary gap, not an emergency — but it means the system's marketed evidentiary rigor is not uniform across the sources it fuses.
4. **Medium — no enforced platform-wide contract for the evidence/confidence/guardrail shape.** A risk for future data source integrations more than a problem with what exists today.

## 7. Architectural Options — presented, not mandated

- Verify the ACLED path question directly; if path A is confirmed unused by any reporting mart, decide explicitly whether to wire it in, replace path B with it, or formally deprecate it. Leaving the ambiguity in place is the one option this review does not recommend.
- Add a lightweight `source_authenticity` or `is_synthetic` flag at `stg.lumen_requests` and propagate it through every table that touches Lumen data, surfaced wherever a Lumen-derived number reaches a page or report.
- Rather than building a new code abstraction, document the evidence/confidence/guardrail shape as an explicit checklist for new data source integrations (a natural fit for `docs/03-development/`) — consistent with the principle of not introducing structure before it is needed. A documented convention is enough to close most of the gap in Section 4 without a rewrite.

## 8. What This Review Does Not Conclude

This review does not conclude that CLIO's methodology is incoherent, nor that the OONI and ACLED-path-A designs should be generalized into a shared abstraction. The strongest parts of the codebase (OONI's dimension-table confidence model, ACLED path A's dual confidence/risk split) are genuinely good, independently arrived-at engineering, and should be preserved as-is. The finding here is narrower: the discipline exists in specific places because specific people wrote it carefully, it does not yet exist as an enforced property of the platform, and one path (Lumen) currently violates it seriously enough to require immediate attention regardless of anything else in this reset.
