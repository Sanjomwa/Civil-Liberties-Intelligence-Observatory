## Cost audit — GCP/BigQuery, performed 2026-07-06

First measured cost pass since ADR-0006's four attribution marts, the protocol-regime mart rewrite, and the daily staleness job shipped. Numbers are **real**, from `region-us-central1.INFORMATION_SCHEMA.JOBS_BY_PROJECT` (180-day retention) and `__TABLES__`, not estimates. Pricing model confirmed **on-demand** (every job carries non-zero `total_bytes_billed`, `reservation_id` = null; us-central1 on-demand = **$6.25/TiB**). All eight datasets are in `us-central1`.

**Headline: absolute spend is tiny — this is a pilot-scale dev project.** Monthly bytes-billed cost: Apr $0.07, May $4.23, Jun $0.04, Jul $2.47 (180-day total ≈ **$6.8**). Warehouse storage ≈ **$1.4/mo** (the `stg` dataset is 64 GiB, 61.5 of it the raw `stg.ooni_measurements` JSON; everything else is sub-3-GiB; the `civil_liberties_prod`/`civil_liberties_staging` raw tables are 0-byte external tables — no duplication waste; `features._dlt_*` bookkeeping is 0 bytes). **The value of this audit is structural, not the current dollar figure** — it identifies where cost *would* scale with data volume, multi-country expansion, or an actually-running `@daily` schedule.

**Ranked cost drivers (90-day bytes-billed, by cost-impact):**

1. **The OONI staging chain — dominant and structural (~$3.2 / 90d, and the only driver that scales badly).** The four observation-extraction assets each re-scan the full ~30 GiB `raw_test_keys` JSON column of `stg.ooni_measurements` on every run, because all are `create+replace` with no incremental filter: `stg.ooni_dns_observations` 181 GiB / 6 runs, `stg.ooni_tcp_observations` 91 / 3, `stg.ooni_http_observations` 60 / 2, `stg.ooni_tls_observations` 60 / 2 (≈30 GiB each every rebuild), plus `stg.ooni_measurements` itself 125 GiB / 11. At the current low rebuild cadence this is ~$3/quarter — but the pipeline's declared schedule is `@daily`; **if that schedule ever actually materializes this chain daily, it is ~4×30 GiB×30 = 3.6 TiB/mo ≈ $22/mo, growing linearly with OONI volume.** This is the single highest-leverage lever. Logged as **TD-58**.

2. **Ad-hoc diagnostic full-scans of `stg.ooni_measurements` (~$1.8 in July alone).** The 61 GiB raw table is unpartitioned and unclustered, so *every* query touching `raw_test_keys` scans the whole column — even `SELECT raw_test_keys ... LIMIT 3` billed 30 GiB. The ~9 largest July SELECTs (30–61 GiB each) were the human OONI-thread investigation queries (TD-47 `lookups` prototyping, `dnscheck` shape checks). Not pipeline cost, but enabled by the missing clustering. Mitigation is part of TD-58 (cluster by `test_name`) plus scan discipline (`--maximum_bytes_billed`, column projection, `TABLESAMPLE`).

3. **The ACLED regime backfill — one-time, but by far the most slot-intensive operation (~31 slot-hours).** 1,547 per-week `MERGE` jobs into `intelligence.acled_pressure_regimes` + `country_initialization_status` (30 GiB billed, but **111,668 + 12,786 slot-seconds**), plus ~12,000 tiny per-week helper SELECTs from the backfill script (these dominate July's 15,578-job count — the count is a quota/concurrency consideration, not a cost one, since they bill almost nothing). The completed backfill is not recurring; steady-state is one week per run via `run_acled_regime_week.py` (TD-38), negligible. **The standing lesson: never casually re-run a full backfill** — it is the most expensive single action available in this repo, in slot-time terms.

4. **Everything else is cents.** All reporting marts, dims, features, and the non-backfill intelligence assets are sub-GiB per materialization. `intelligence.acled_pressure_regimes` steady-state, `features.protocol_daily_signals` (22 GiB / 16 runs — the OONI-derived one, mid-size), and the attribution marts are all immaterial.

**Streamlit dashboard cost is negligible and needs no change.** Every `services/marts.py` query reads a small reporting/marts table (all ≤ a few MiB — `mart_political_stress_windows` 761 rows, etc.) and every one filters `WHERE date_key/measurement_date BETWEEN …`; `run_query` is wrapped in `@st.cache_data(ttl=3600)`. Observed dashboard/SELECT scans against reporting tables bill 0.00 GiB. The date filters would benefit from partitioning *if* these tables were large, but they aren't — so partitioning the reporting layer is not worth the complexity today (noted, not recommended).

**Partitioning/clustering inventory:** **zero tables in the pipeline are partitioned or clustered.** For the tiny reporting/marts/dims tables this is correct (partition metadata overhead would exceed the benefit). It matters only for the OONI chain — see TD-58.

**Materialization-strategy inventory:** 46 of 47 SQL assets use `create+replace`; only `intelligence.acled_pressure_regimes` uses `merge` (it must — the stateful regime engine). For every small table, `create+replace` is the right, simplest choice. The one genuine strategy/size mismatch is the OONI observation chain (large source, append-mostly data, full recompute every run) — TD-58.
