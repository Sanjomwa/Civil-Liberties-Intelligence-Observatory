## Narrative decision log

Unlike the ADR index above, this section exists to preserve *why* a conclusion was reached, including conclusions that haven't yet become an implementation, so a future contributor understands the reasoning, not just the outcome. Entries are dated and appended in order; none are edited retroactively — a changed conclusion gets a new entry that references the one it supersedes.

**2026-07-05 — Four architectural questions were separated out of "Path A vs Path B."** Direct investigation (matching this project's now-repeated pattern of checking a claim's actual shape before acting on it) found "should Path A replace Path B" was never a single decision: Path A interprets one evidence source (ACLED); Path B performs cross-source fusion. The real open questions are narrower: what CLIO's flagship signal should look like given Path A's weekly-categorical output versus Path B's daily-continuous one; what happens to Fusion's legal and platform inputs, which have no Path-A-equivalent treatment at all; the per-country backfill cost of Path A's pattern; and whether the evidence gathered so far (the Finance Bill 2024 comparison, TD-40/TD-02/TD-01/ADR-0001's fixes) is sufficient to act on now versus needing a longer observation period. No decision was forced on any of these; they were named as the actual decomposition of a question that had been treated as one big binary choice.

**2026-07-05 — A four-layer mental model (Evidence, Source Intelligence, Fusion, Presentation) was evaluated against the real codebase and found genuinely present, not invented.** OONI and ACLED path A each independently implement `Observation → Interpretation → Confidence → Inference` across real, separate Bruin assets — two concrete instances, meeting this project's own bar for when a pattern is allowed to become a named principle. Two real gaps were found rather than assumed away: Lumen and Google Transparency have no equivalent Source Intelligence layer (one because its data is fabricated, the other because it has never been audited). One real naming problem was found: `marts.facts` currently conflates true cross-source fusion (`fact_country_pressure_daily`) with single-source facts that merely live in the same folder (`fact_takedown_activity`, `fact_asn_repression_index`). The model was adopted as documentation-level vocabulary only — no physical refactor, no shared interface/framework across sources, since only two of four sources currently justify the pattern and two do not yet exist in a state that could conform to one. The one concrete, low-risk implementation question this reframing produced: should `fact_country_pressure_daily`'s `conflict_pressure_score` be derived from `intelligence.acled_pressure_regimes` instead of `stg.acled_conflict_events` directly — not yet investigated or decided.

**2026-07-05 — CLIO's two primary users were derived, not assumed, from the Project Zero Review's own buyer research, and narrowed to the smallest set CLIO can serve well today.** Working backward from user to decision to information to evidence (rather than starting from architecture), the investigative journalist and the NGO/human-rights-documentation buyer (including legal teams and human rights commissions, whose decision shape is identical) were confirmed as the smallest useful set — matching the Zero Review's own finding that these two "receive the most value per dollar" because they cannot produce this evidence themselves and have an immediate, concrete need for a citable finding. Election observer missions, academic researchers, and corporate/government buyers were explicitly named as out of scope for now, each for a stated reason (forward-looking/operational decision shape mismatched to CLIO's retrospective methodology; low urgency; missing sales/legal/security infrastructure and a real mission-drift risk), not silently deprioritized. Both primary users' actual information requirement converged independently on the same conclusion the architecture-first analysis reached separately: an attributed breakdown of *why* pressure is elevated, not a single composite score — confirming the Project Zero Review's own still-unbuilt Recommendation #4 (the pressure-attribution decomposition) as the correct next priority, reached twice, from two independent directions.

**2026-07-05 — CLIO's evidence architecture was defined: OONI and ACLED confirmed as foundation datasets, explicit admission criteria derived for any future source, and six real candidate datasets evaluated against them.** See ADR-0003 for full detail. Summary conclusion: Access Now's Shutdown Tracker Optimization Project and CPJ's case-level killed/attacked journalist data are recommended as Core (not yet ingested); IODA and CPJ's imprisonment-census data as Secondary (corroboration only); Freedom House as Optional (manual annual citation only, never pipeline ingestion); CIPESA rejected as a pipeline source (its real value is manual citation in report narrative, outside the SQL pipeline entirely); Google Transparency and further platform-transparency sources deliberately left unresolved pending an audit that has not yet been performed. The explicit intent, per instruction, was a small number of excellent sources rather than a long list of mediocre ones — two Core recommendations came out of six candidates surveyed.

**2026-07-05 — the evaluation framework itself was refined: chain-strengthening and vector-opening are two distinct, non-substitutable reasons to admit an evidence source, not one test.** Scrutinizing *why* STOP scored Core (not just that it did) found its value is disproportionate specifically because it attaches to OONI and ACLED's existing relationship — turning a temporal correlation between a technical signal and a conflict signal into an attributed official act. That is chain-strengthening value. CPJ's case-level data was re-examined against the same lens and found to work differently: it doesn't reinforce any existing relationship, it opens a decision-relevant question (was the press itself directly targeted) that no existing source asks at all — vector-opening value. Evaluating all future candidates only by "does it strengthen what we already have" would systematically undervalue genuinely new evidence like CPJ's in favor of evidence that merely reinforces existing strength. Both tests are now retained explicitly, not collapsed into one, and ADR-0003 records this refinement rather than silently superseding the original framing.

**2026-07-05 — Google Transparency was studied with the same rigor as STOP and CPJ, rather than left deferred, and required a fifth classification tier CLIO's evidence architecture hadn't yet named.** Its real grain (semi-annual, country/product/reason aggregate, with a request count and a compliance-rate figure) does not support incident-level interpretation without fabricating precision the source doesn't have — the same category of mistake this project has now caught and corrected at TD-40, TD-02, and ADR-0002 step 5. But it is real, structured, independently-provenanced data, unlike CIPESA's qualitative reporting, and its compliance-rate figure is a genuinely new evidentiary category (consequential evidence — did applied pressure actually succeed) that nothing else in CLIO's evidence set answers. Neither Core, Secondary, Optional, nor Rejected fit it honestly, so a fifth tier, **Contextual/Backdrop**, was added: a legitimate pipeline input, computed and reported strictly at its own six-month grain, feeding report narrative rather than any incident-level score or confidence claim, and explicitly barred from being broadcast down to a finer grain than it actually has.

**2026-07-05 — Lumen was formally benched, and the pilot product was explicitly distinguished from CLIO's long-term evidence architecture.** See ADR-0004. Benching is deliberately not deletion and not a "known limitation" note: the architecture keeps its conceptual support for legal-pressure evidence as a category (schema, composite-score term, provenance-flag machinery all remain), while the pilot honestly acknowledges it has no production-quality source for that category today. This surfaced one concrete question deliberately left open rather than decided in the same breath: whether benching should extend to re-normalizing `composite_pressure_score`'s weights to exclude Lumen's term entirely, since today the score still arithmetically includes the fabricated input at full weight even though everyone agrees it isn't evidence — a disclosure banner discloses the problem without the arithmetic actually reflecting the decision. The pilot/architecture distinction itself was named explicitly: the Kenya pilot using only currently-qualifying sources is a scoping decision, not a ceiling — a future customer needing a legal- or platform-pressure-centered product should be served from whatever sources meet CLIO's bar at that time, without that implying today's Lumen placeholder was ever adequate for it.

**2026-07-05 — ADR-0004's open composite-score question was resolved by investigation, not intuition: Option B (drop Lumen's term, renormalize to conflict 0.75 / platform 0.25).** A full inventory of every computation and consumption site found two things not previously on record: `protocol_repression_correlation_mart`'s `z_pressure` is computed via a global, unpartitioned window function, meaning any reweighting shifts the normalization baseline for every row in the table's history, not just Lumen-affected ones; and `mart_political_stress_windows` recomputes a second, different formula under the identical column name `composite_pressure_score`, which is the value the dashboard's flagship KPI actually reads, not the fact table's raw one (logged as TD-45). The decisive finding for choosing B over C: `legal_pressure_is_synthetic = FALSE` never means "real data present" in the live data today, only "no row exists," and `legal_pressure_score` is independently 0 on those same dates — so a conditional/dynamic renormalization would be arithmetically identical to a static one on every row that currently exists, meaning Option C's added complexity would be pure future-proofing for a case that doesn't exist yet, which doesn't meet this project's own two-concrete-cases bar. A correction to ADR-0002's own language was also produced: the Finance Bill window's "floor propped up ~5-6" description turns out to be partly backwards on 9 specific dates, where Lumen's *absence* (not its presence) produced a local minimum via `COALESCE(...,0)` — a minor nuance, not a reversal. Two adjacent findings were deliberately not folded into this decision: the `pressure_level` thresholds were never empirically re-derived and 29.6% of days now change category, worth its own investigation (TD-44); and the naming collision above (TD-45). Implementation approved, not yet executed.

**2026-07-05 — ADR-0004's composite-score question shipped (Option B), verified against real materialized data, with two adjacent findings logged and one disclosure gap surfaced.** Implementation confirmed the investigation's prediction exactly: a pre-materialization simulation matched the actual post-`bruin run` output on every metric checked. Real empirical shift: mean `composite_pressure_score` moved from 4.6336 to 4.1791; 225/761 days (29.6%) changed `pressure_level` category; the Finance Bill 2024 flagship day stayed `ELEVATED` under both formulas, unaffected. TD-08's golden-file tests were re-run and confirmed unaffected, as predicted. Two new items were logged rather than folded into the fix: TD-44 (the `pressure_level` thresholds were never empirically re-derived, and a 29.6% category shift is real reason to check them) and TD-45 (a naming collision between `fact_country_pressure_daily.composite_pressure_score` and `political_stress_windows_mart`'s own differently-computed value of the same name — the dashboard's flagship KPI reads the mart's recomputed value, not the fact table's raw one). ADR-0002's own "5-6 floor" language was annotated, not rewritten, once re-verification found it was already only approximately true before this change (one day, 2024-05-25, was already `MODERATE` pre-reweighting, since Lumen had no row that day). A small documentation drift was also caught and corrected while syncing this record: an earlier relay had mistranscribed TD-43's file location as `int.ooni_experiment_results.sql:267`; Claude Code's direct read of the live repo during this work confirmed the real location is `int.acled_event_classification.sql:267`, and the record was corrected to match rather than left disagreeing with the live repo.

One real disclosure gap surfaced during implementation was resolved the same day it was found (TD-46, commit `6875be7`): `synthetic_data_notice`'s banner on pages 1, 6, and 7 still fired whenever `legal_pressure_is_synthetic` was true, overclaiming that the displayed KPI was tainted after Lumen no longer fed the composite score. Investigated before fixing, per standing discipline: every service function and render call on all three pages was checked directly, confirming `legal_pressure_score` isn't fetched on any of them and `legal_pressure_is_synthetic`, while fetched, was never actually rendered. Since nothing Lumen-related reaches the user on these pages, all three banners were removed outright rather than reworded, verified via headless-Chromium screenshots against fresh materialized data. The shared `synthetic_data_notice()` component itself was deliberately left in place, unused but intact, since it's generic and reusable, not specific to this one case.

**2026-07-05 — STOP and CPJ's real ingestion feasibility was investigated against fetched data, not documentation claims, and CPJ's classification was materially revised as a result.** See ADR-0003's addendum for full detail. STOP: real CSV access confirmed (317 live rows fetched via Google Sheets' export endpoint), CC BY 4.0, event-level grain confirmed, and a concrete triangulation finding — the fetched Kenya rows independently cite AS37061/AS36926/AS33771, the exact three ASNs already tracked as `MAJOR_KENYA_PROVIDER` in CLIO's own `asn_behavior_profile_mart`. CPJ: licensed CC BY-NC-ND 4.0 (Attribution-NonCommercial-NoDerivatives), confirmed directly from CPJ's own methodology page — a real conflict with CLIO's business model (a commercial, derivative, fused product), not known when CPJ was classified Core. CPJ also has no official export or documented API; the only access path found was an undocumented internal REST endpoint discovered by inspecting network traffic, a real operational risk independent of the license question. CPJ's classification is downgraded from Core to blocked-pending-legal-resolution — no further technical work should happen until CPJ (`press@cpj.org`) has been asked directly whether CLIO's specific intended use is permitted. STOP's classification held and sharpened; it is now the recommended next real ingestion candidate. A documentation gap was also surfaced: the "Evidence / Source Intelligence / Fusion / Presentation" model referenced repeatedly in this project's recent reasoning has never actually been written down anywhere — it exists only as an informal mapping reconstructed each time it's needed, and now has enough repeated real use (this project's own two-concrete-cases bar) to justify writing it down properly.

**2026-07-05 — a diagnostic-only investigation traced whether OONI's app-identity and protocol-layer-cause detail survive CLIO's pipeline, prompted by a direct question about attributing findings to specific apps (Telegram, WhatsApp, Signal) and specific protocol layers (DNS/TCP/TLS/HTTP).** No code changed. The question was well-founded: OONI's raw measurements are structured exactly this way (distinct `test_name` values per app, plus per-layer anomaly fields), so this was a real gap to check, not a speculative enhancement. Findings, traced file-by-file from raw to reporting against live BigQuery data: `web_connectivity` (the generic web-blocking test most OONI narratives assume as default) and `facebook_messenger` were never ingested at all — the six real test types are `dnscheck`, `whatsapp`, `telegram`, `psiphon`, `signal`, `tor`. Of those, `dnscheck` and `tor` (~85% of all raw rows ingested for Kenya) are silently dropped at the staging observation layer's `UNNEST` extraction and never reach any interpreted table — logged as TD-47, the most severe of six new items, since its cause (a JSON key-path mismatch for two specific test types) hasn't yet been diagnosed as fixable-by-design or a deeper ingestion bug. For the four test types that do survive: raw app identity (`test_name`) is preserved intact through raw, staging, intermediate, and one live fact table, then collapsed to a coarse 3-bucket `test_family` at `features/protocol_daily_signals.sql` (TD-48), and that coarser dimension is dropped a second time before reaching any reporting mart Streamlit actually reads (TD-49) — confirmed via `grep`, `test_name`/`test_family` appear nowhere in `streamlit/`. Separately, cause-level protocol detail (e.g. `dns.nxdomain` vs. `tcp.rst` vs. `tls.timeout`, not just "which layer") survives as explicit count columns through `features/protocol_daily_signals.sql` and is dropped at `intelligence/protocol_signal_regimes.sql`, the exact point past which only a blended stress score is knowable (TD-50). One genuinely good finding: `marts.fact_platform_blocking_summary` already carries full, uncoarsened `test_name` and `protocol` resolution at monthly grain — the one asset in the whole chain where this question is already answered structurally — but it has zero consumers anywhere (TD-51, the same class of finding as the Step 0 ACLED path A reporting gap), and its own attempted cause-breakdown columns have a live, verified bug making them permanently zero regardless (TD-52). Net assessment: restoring app-level, protocol-layer-attributed censorship findings to the dashboard is not a new-ingestion problem for four of six test types — it's substantially a "finish and wire in what's already built" problem (TD-51 fixed via TD-52, plus TD-48/49's dimension restored at the reporting layer), with TD-47's dnscheck/tor exclusion as a separate, larger, and more urgent question about a near-total gap in the DNS-censorship test type specifically. No implementation decision made yet; this entry documents the diagnostic only, per explicit instruction to investigate before proposing fixes.

**2026-07-05 — TD-47 was diagnosed: `dnscheck` and `tor`'s exclusion is not one bug, but two structurally different OONI test designs never accounted for by the staging layer's shared four-table model.** No code changed. Direct inspection of live raw rows (not documentation) confirmed the hypothesis from the prior entry: `whatsapp`/`telegram`/`signal`/`psiphon` all share OONI's flat, per-experiment `measurex`-style shape (top-level `queries`/`tcp_connect`/`tls_handshakes`/`requests` arrays), which is exactly what the four staging observation assets' fixed JSONPaths expect — this is why those four work and `dnscheck`/`tor` silently don't, rather than a general extraction bug. `dnscheck` turns out to be two things bundled under one test name: a `bootstrap` subset (one system-resolver DNS lookup, cheaply recoverable via a corrected fixed path, leaf-compatible with existing classification logic) and a `lookups` subset (per-resolver DNS censorship-resistance testing — arguably the actual point of the test) that cannot be reached by any fixed BigQuery JSONPath at all, since its keys are runtime-determined resolver URLs; recovering it requires ingestion-time flattening in Python, not a staging SQL fix. `tor` is not a path problem at all — it measures reachability of named circumvention infrastructure (relays, directory authorities, bridges) via pre-aggregated summary counters, answering a different evidentiary question ("is the circumvention tool reachable") than every other OONI test CLIO ingests ("is protocol layer X blocked for app Y"), with no equivalent column anywhere in the current schema. This reframes TD-47 from one bug into three independent next steps of very different size and character (a cheap JSONPath fix, a real ingestion-side feature, and a new-asset-plus-scope question) — logged as such in `technical-debt-inventory.md` rather than collapsed into a single "fix the UNNEST" action. No implementation decision made yet.

**2026-07-05 — decided, in the project owner's absence per their explicit request: the manner of tackling the TD backlog going forward is tiering by cost and blast radius within a coherent thread, not discovery order or raw severity label.** The backlog is growing faster than it's closing (52 items logged, most from diagnostic investigation rather than implementation), which makes an explicit operating policy worth stating rather than re-deciding ad hoc each time. Policy: (1) group items that came from the same investigation thread rather than treating each TD number as an independent unit of work — TD-47 through TD-52 are one thread (OONI app/protocol attribution), not six unrelated tickets; (2) within a thread, sequence cheap+isolated fixes first (Tier 1: no design judgment needed, small blast radius, often literally "finish something already built" — e.g. TD-51/52's already-materialized-but-unwired mart), then bounded design decisions (Tier 2: a real choice exists but the scope is contained), then genuine investigations or features (Tier 3: needs its own scoping pass, possibly its own ADR, before implementation starts); (3) resist starting Tier 3 work inside a thread until Tier 1 has shipped and been observed for a while, since Tier 1 often changes what Tier 3 actually needs (e.g. whether TD-48/49's app-family restoration is even necessary depends on whether TD-51's already-full-resolution panel turns out to satisfy the need on its own); (4) tor's inclusion (TD-47's third sub-item) is treated as a scope decision routed through ADR-0003's existing evidence-admission reasoning, not a default "since the data exists, ingest it" — consistent with this project's repeated finding that dataset availability and dataset value are different questions. Applied immediately to the OONI thread in `implementation-roadmap.md`'s new Step 3b. This is a durable policy for future TD growth, not a one-time sequencing decision for this thread alone.

**2026-07-05 — a still-open reflection, not a decision: is CLIO's long-term identity a fixed product or an evidence engine assembling specialized products per customer?** Raised as a genuine question, not resolved here. The reasoning for taking it seriously is stronger than an abstract industry claim about code generation becoming commoditized: every piece of real value produced this week (TD-40, TD-02, TD-01, ADR-0001) came from correctly diagnosing what evidence actually means, not from writing new capability — direct, first-party evidence that this project's actual scarce resource is epistemic quality, not engineering throughput. The question was deliberately split into two scopes rather than answered as one. The narrow version — assembling different evidentiary emphases per customer within the existing Kenya civil-liberties mission — looks likely correct and low-risk, and is substantially already implied by the pressure-attribution decomposition work already scoped (Project Zero Review Recommendation #4, still unbuilt): splitting a composite score into named, sourced drivers is most of what "assemble a specialized report" requires. The broad version — CLIO's methodology as a generalizable product across domains or customers entirely outside Kenya civil liberties — is not a new idea; it is the "Governance Tooling" model the Project Zero Review already named independently (Section 3.10) and explicitly gated behind the Constitution's own empirical research programme producing supporting evidence first. Nothing found this week loosens that gate. If anything it tightens it: foundational bugs were still being found in the one, single, concrete instance of this methodology as recently as this week, which argues for proving that instance fully before generalizing from it, not after.
