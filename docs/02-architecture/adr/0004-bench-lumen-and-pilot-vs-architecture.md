# ADR-0004: Formally Bench Lumen; Distinguish the Pilot Product from the Long-Term Evidence Architecture

Status: Accepted (decision recorded; one implementation question deliberately left open, see Consequences)
Date: 2026-07-05

## Context

TD-01 (resolved 2026-07-05) made Lumen's synthetic-data status honestly visible throughout the pipeline and on every Streamlit page that touches it — a real, necessary fix. But a provenance flag answers "is this real," not "what should CLIO's relationship to this data be going forward." Left unaddressed, Lumen sits in an ambiguous middle state: not fixed, not removed, quietly still contributing 25% of `composite_pressure_score`'s weight with a warning attached, which reads as a permanent apology rather than a deliberate engineering stance.

Separately, ADR-0003's evidence architecture study raised a related question by implication: if CLIO's real differentiator is evidentiary quality rather than dataset quantity, what does CLIO do when a future customer needs a report centered on exactly the kind of evidence (legal or platform pressure) that doesn't currently meet that bar? The two questions are connected: how CLIO treats a currently-inadequate evidence source now shapes what kind of product CLIO can credibly become later.

## Decision

**Lumen is formally benched, not deleted and not silently tolerated.** The distinction is deliberate: CLIO's architecture continues to conceptually support legal-pressure evidence as a category — the schema, the CTE, the composite-score term, and the provenance-flag machinery TD-01 built all remain in place — but the current pilot explicitly acknowledges it does not presently have a production-quality legal-pressure dataset that meets the admission criteria ADR-0003 defined (independent provenance, in particular — a fabricated dataset has none). This is documented here as a deliberate engineering decision, not tracked as a missing feature or an unresolved bug: there is nothing to "fix" about Lumen until a real data source exists to replace it; the current implementation was always a placeholder, and TD-01's job was to stop it from being mistaken for anything else.

**The pilot product is intentionally narrower than CLIO's long-term evidence architecture.** The Kenya civil-liberties pilot uses whichever sources currently meet ADR-0003's bar — OONI and ACLED today, with Access Now's STOP and CPJ's case-level data as the next real candidates. That is a scoping decision for the pilot, not a ceiling on the architecture. If a future NGO, journalist, researcher, or institution needs a specialized product centered on platform moderation or legal takedown activity specifically, CLIO should be capable of assembling that product from whichever evidence sources satisfy its standards at that time — which may or may not include a real Lumen-equivalent by then. Benching today's Lumen does not bench the category of evidence it represents; it benches this specific, currently-inadequate implementation of it.

## Consequences

Documentation and product framing should reflect "benched" language going forward, not "known limitation" language — the two read differently to a future contributor and to a customer, and the distinction is the point of this ADR.

**One concrete implementation question is raised here but deliberately not decided:** today, `fact_country_pressure_daily`'s `composite_pressure_score` still arithmetically includes Lumen's fabricated contribution at its full weight (0.25), with only a disclosure banner attached — the warning tells a reader some of the number is fake, but the number itself does not stop incorporating it. Whether "benching" should extend to re-normalizing the composite score's weights to exclude Lumen's term entirely (conflict and platform only) until real data exists, rather than continuing to average in a value both this ADR and TD-01 agree isn't evidence, is a real question this decision surfaces. It is not resolved here — it should be investigated with the same discipline as every other change this project has made this week (verify the actual consumer impact of re-weighting before implementing it, since `mart_political_stress_windows` and `protocol_repression_correlation_mart` both pass the current composite score through, and a re-weighted score is a real, visible behavior change to anyone already citing today's numbers).

This ADR does not authorize building a generalized "assemble from any qualifying source" product configuration system now. Exactly one specialized-product need has ever been named (a hypothetical legal/platform-focused report for a future customer), which does not meet this project's own two-concrete-cases bar for building a shared abstraction. What this ADR does authorize is treating the *pilot* and the *architecture* as two different scopes when reasoning about future work, so that a source's unsuitability for today's pilot (like Lumen's) is never mistaken for unsuitability for CLIO's architecture as a whole.
