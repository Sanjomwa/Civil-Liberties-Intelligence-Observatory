# ADR-0004: Formally Bench Lumen; Distinguish the Pilot Product from the Long-Term Evidence Architecture

Status: Accepted (decision recorded; one implementation question deliberately left open, see Consequences)
Date: 2026-07-05

## Context

TD-01 (resolved 2026-07-05) made Lumen's synthetic-data status honestly visible throughout the pipeline and on every Streamlit page that touches it — a real, necessary fix. But a provenance flag answers "is this real," not "what should CLIO's relationship to this data be going forward." Left unaddressed, Lumen sits in an ambiguous middle state: not fixed, not removed, quietly still contributing 25% of `composite_pressure_score`'s weight with a warning attached, which reads as a permanent apology rather than a deliberate engineering stance.

Separately, ADR-0003's evidence architecture study raised a related question by implication: if CLIO's real differentiator is evidentiary quality rather than dataset quantity, what does CLIO do when a future customer needs a report centered on exactly the kind of evidence (legal or platform pressure) that doesn't currently meet that bar? The two questions are connected: how CLIO treats a currently-inadequate evidence source now shapes what kind of product CLIO can credibly become later.

## Decision

**Lumen is formally benched, not deleted and not silently tolerated.** The distinction is deliberate: CLIO's architecture continues to conceptually support legal-pressure evidence as a category — the schema, the CTE, the composite-score term, and the provenance-flag machinery TD-01 built all remain in place — but the current pilot explicitly acknowledges it does not presently have a production-quality legal-pressure dataset that meets the admission criteria ADR-0003 defined (independent provenance, in particular — a fabricated dataset has none). This is documented here as a deliberate engineering decision, not tracked as a missing feature or an unresolved bug: there is nothing to "fix" about Lumen until a real data source exists to replace it; the current implementation was always a placeholder, and TD-01's job was to stop it from being mistaken for anything else.

**The pilot product is intentionally narrower than CLIO's long-term evidence architecture.** The Kenya civil-liberties pilot uses whichever sources currently meet ADR-0003's bar — OONI and ACLED today, with Access Now's STOP and CPJ's case-level data as the next real candidates. That is a scoping decision for the pilot, not a ceiling on the architecture. If a future NGO, journalist, researcher, or institution needs a specialized product centered on platform moderation or legal takedown activity specifically, CLIO should be capable of assembling that product from whichever evidence sources satisfy its standards at that time — which may or may not include a real Lumen-equivalent by then. Benching today's Lumen does not bench the category of evidence it represents; it benches this specific, currently-inadequate implementation of it.

## Consequences

Documentation and product framing should reflect "benched" language going forward, not "known limitation" language — the two read differently to a future contributor and to a customer, and the distinction is the point of this ADR.

**One concrete implementation question is raised here but deliberately not decided:** today, `fact_country_pressure_daily`'s `composite_pressure_score` still arithmetically includes Lumen's fabricated contribution at its full weight (0.25), with only a disclosure banner attached — the warning tells a reader some of the number is fake, but the number itself does not stop incorporating it. Whether "benching" should extend to re-normalizing the composite score's weights to exclude Lumen's term entirely (conflict and platform only) until real data exists, rather than continuing to average in a value both this ADR and TD-01 agree isn't evidence, is a real question this decision surfaces. It is not resolved here — it should be investigated with the same discipline as every other change this project has made this week (verify the actual consumer impact of re-weighting before implementing it, since `mart_political_stress_windows` and `protocol_repression_correlation_mart` both pass the current composite score through, and a re-weighted score is a real, visible behavior change to anyone already citing today's numbers).

**RESOLVED, 2026-07-05: Option B shipped (drop Lumen, statically renormalize to conflict 0.75 / platform 0.25).** An investigation preceding this fix evaluated three options (status quo; static renormalization; a conditional per-row renormalization keyed on `legal_pressure_is_synthetic`) and found the third not currently justified: every date with a Lumen row today has `legal_pressure_is_synthetic = TRUE` (the flag's `FALSE` state means "no Lumen row that date," not "real Lumen data"), so a conditional design would execute identically to Option B on all 761 live rows and buy nothing today — its only value is prospective, for a future real Lumen-equivalent, and is deferred until one exists.

Implemented in `fact_country_pressure_daily.sql`: `composite_pressure_score = conflict_pressure_score*0.75 + platform_pressure_score*0.25` (legal term removed, not multiplied by zero); `pressure_level`'s three CASE branches now reference the `composite_pressure_score` column (computed in its own `composite` CTE) instead of re-deriving the formula inline three times, closing a pre-existing repetition debt in the same pass. `political_stress_windows_mart.sql` and `protocol_repression_correlation_mart.sql` needed no logic changes (pure passthrough of the fact table's columns) — only header/inline comments updated and `reporting_version` bumped (`political_stress_windows_mart_v4`→`v5`, `protocol_repression_correlation_mart_v3`→`v4`) to signal the methodology change. `legal_pressure_score`, `legal_pressure_is_synthetic`, and the `lumen` CTE all remain untouched in the schema, per this ADR's "benched, not deleted" decision.

**Before materializing**, given `protocol_repression_correlation_mart`'s `z_pressure` is a global, unpartitioned `AVG/STDDEV_SAMP OVER ()` normalization (re-basing every row in the table's full history against a shifted mean/stddev, not just Lumen-affected rows), an ad-hoc query reran the mart's full CTE chain against live source data with the new formula substituted in, before touching the live table. Verified before vs. after (3,044 rows, live Kenya data):

| Metric | Before (0.60/0.25/0.15) | After (0.75/0/0.25) |
|---|---|---|
| `fact_country_pressure_daily` mean / stddev `composite_pressure_score` | 4.6336 / 0.3194 | 4.1791 / 0.3058 |
| `pressure_level` distribution (761 days) | 748 ELEVATED, 13 MODERATE | 537 ELEVATED, 224 MODERATE (225 days / 29.6% changed category — logged as **TD-44**, thresholds not re-derived) |
| `correlation_state` (3,044 protocol-days) | 68 INSUFFICIENT_HISTORY, 1,488 WEAK_OR_NO_RELATIONSHIP, 1,488 ZERO_VARIANCE_WINDOW | 68 / 1,487 / 1,488, plus 1 row newly `MODERATE_RELATIONSHIP` |
| `alignment_state` | 2,245 NO_CLEAR_ALIGNMENT, 606 PRESSURE_ONLY, 193 PROTOCOL_DIVERGENCE | 2,307 / 537 / 199, plus 1 row newly `INVERSE_MOVEMENT` |
| `divergence_state` | 86 HIGH, 2,723 LOW, 235 MODERATE | 95 HIGH, 2,651 LOW, 298 MODERATE |

The ad-hoc simulation matched the actual post-`bruin run` materialized values exactly on every row above — no surprises between prediction and reality. The shift is real but bounded (roughly 2–3% of protocol-days change `alignment_state`/`divergence_state` category; `correlation_state` changes by exactly 1 row), consistent with re-basing a global normalization after removing a term that averaged ~31% of the composite's mass.

**Finance Bill 2024 window (2024-05-11 to 2024-07-13, 64 days), re-verified against the actual materialized table:** the flagship day, 2024-06-22 (Parliament stormed), stays `ELEVATED` under both formulas (5.4292 → 5.0989) — the specific comparison ADR-0002 draws attention to (path A correctly flags this week `CRISIS` while path B never reaches `SEVERE`) is unaffected. Across the full window: 57/64 days stay `ELEVATED`, 1/64 stays `MODERATE`, and 6/64 move `ELEVATED`→`MODERATE`. This is a **correction to this ADR's own prior framing**, found only by checking rather than assuming: the window was not uniformly `ELEVATED` even before this change (2024-05-25 was already `MODERATE`, since `legal_pressure_score` happened to be `0` that day — no Lumen row existed for it), so "the window's category is unaffected" is true for the flagship day specifically but not for all 64 days in it. See the annotation added to ADR-0002 for the full detail.

Verification performed: `bruin validate` (56/56 clean, before and after), `bruin run` materialization of all three affected assets (`fact_country_pressure_daily`, `political_stress_windows_mart`, `protocol_repression_correlation_mart`), TD-08's golden-file tests re-run with `RUN_BIGQUERY_TESTS=1` (6 passed — confirmed unaffected, as predicted, since they assert only `intelligence.acled_pressure_regimes` fields, never `fact_country_pressure_daily`'s). `streamlit/pages/8_Methodology_Statistical_Guardrails.py`'s hardcoded "25% of `composite_pressure_score`" warning rewritten to describe exclusion rather than weighted inclusion. TD-01's `technical-debt-inventory.md` row and this ADR's own prior paragraph in ADR-0002 are annotated (not rewritten) as describing the pre-reweighting state. TD-44 (pressure_level thresholds not re-derived) and TD-45 (the pre-existing `composite_pressure_score` naming collision between this fact table and `political_stress_windows_mart`'s own recomputed value of the same name, discovered during this investigation) are logged in `technical-debt-inventory.md`, neither fixed here.

This ADR does not authorize building a generalized "assemble from any qualifying source" product configuration system now. Exactly one specialized-product need has ever been named (a hypothetical legal/platform-focused report for a future customer), which does not meet this project's own two-concrete-cases bar for building a shared abstraction. What this ADR does authorize is treating the *pilot* and the *architecture* as two different scopes when reasoning about future work, so that a source's unsuitability for today's pilot (like Lumen's) is never mistaken for unsuitability for CLIO's architecture as a whole.
