# ADR-0002: Wire the sophisticated ACLED regime engine (Path A) into reporting

Status: Accepted
Date: 2026-07-04

## Context

Step 0 of the Implementation Roadmap confirmed, independently, twice — once against a pre-commit snapshot of the repository and once against the live Codespace repo via Claude Code, with identical results — that the dashboard's national pressure score is built entirely from ACLED path B (`stg.acled_conflict_events` → `marts.fact_country_pressure_daily` → `reporting.mart_political_stress_windows`), and does not consume `intelligence.acled_pressure_regimes` (path A) or `intelligence.country_readiness` anywhere. Path A is the more methodologically mature of the two: it assigns a dual `classification_confidence`/`methodology_risk_level` split, applies per-family statistical guardrails, and performs stateful regime classification, none of which path B does. See `methodology-consistency-review.md` Sections 2 and 6, and `technical-debt-inventory.md` TD-03.

Three options existed: wire path A in, formally deprecate it, or defer the decision while hardening path B. Given the product strategy explicitly sells confidence-qualified findings to legal and journalistic buyers, the methodological gap between what is sold and what is actually computed was judged too consequential to leave unresolved or to resolve by discarding the more rigorous path.

## Decision

Wire path A into the reporting layer, replacing path B as the source of the dashboard's national ACLED pressure signal. This is not a single change — it requires, in order:

1. **Done.** Establish the actual current state of `intelligence.acled_pressure_regimes` and `intelligence.country_initialization_status` before anything else — confirmed a partial backfill existed (15/1,523 weeks), not the zero or complete state either extreme would have suggested.
2. **Done, 2026-07-04.** Complete or resume the historical backfill via `Bruin/scripts/historical_initializer/backfill_acled_pressure_regimes.py` for whatever history is missing, since path A's stateful classification requires sequential, committed history to produce meaningful regime labels rather than defaulting every week to `is_first_observation_week`. Kenya is now `COMPLETE`: 1,523/1,523 weeks (1997-01-11 → 2026-03-14), 0 errors across five batches. This produced the first real evidence for this ADR's premise: the classifier cleanly detected the Jan 2008 post-election violence, the Aug 2022 election, and — most importantly for the product — the Finance Bill 2024 protests, with a fresh CRISIS transition landing exactly on the June 22-28 week Parliament was stormed (protest/suppression bands both SEVERE, HIGH confidence, no lag), sustained through Saba Saba, clean reversion after. It also produced two plausible true-negatives (2010 referendum, Sept 2017 court annulment) and confirmed TD-05's confidence-decay question is most likely a data-sparsity artifact of pre-2020 ACLED coverage, not a structural flaw (0 `INSUFFICIENT_DATA` weeks from 2020 onward, across 111 non-STABLE weeks). See the candidate list below for what's usable as a golden file and what still needs care.
3. **Done, 2026-07-04.** Add the runtime precondition guard to `intelligence.acled_pressure_regimes.sql` (TD-04) before this asset becomes load-bearing for a live, sold product — today a precondition violation corrupts state silently, which was tolerable while the asset was unconsumed and is not tolerable once it feeds a client-facing number. The precondition actually lives under the file's "EXECUTION CONTRACT (AUTHORITATIVE)" comment block (not "ARCHITECTURE FREEZE"/BLOCKER-002 as earlier documentation here mistakenly said — corrected). Design: Bruin's native `custom_checks` run *after* asset execution, too late to prevent a corrupting MERGE, so the guard is instead a new upstream Bruin asset (`intelligence.acled_pressure_regimes_precondition_check`) that calls `ERROR()` on violation and is added to the regime asset's `depends:` list — Bruin's DAG then refuses to invoke the regime asset at all on failure, rather than letting it run and corrupt state. Applied and verified against the real repo: `bruin validate` (57/57 assets clean) and `bruin lineage` (new check confirmed upstream of the regime asset). Committed `2080b64`, purely additive — no change to the existing MERGE or materialization logic, and the completed 1,523-week Kenya backfill is untouched.

   **This design work surfaced a more urgent, separate finding — see TD-38, now resolved.** `Bruin/pipeline.yml`'s documented steady-state default (`target_week=""`) silently violated this same EXECUTION CONTRACT, because the upstream `features.acled_pressure_signals` asset recomputes the entire country history every run (`strategy: create+replace`, no incremental filter) rather than just the newest week. Fixed via a new `Bruin/scripts/steady_state/run_acled_regime_week.py`, which resolves the single next unprocessed week itself and always passes it explicitly to Bruin, never `""` — reusing the backfill script's own committed-week-lookup logic rather than duplicating it, and no-op'ing cleanly (exit 0, no side effects) when no new week is available. Verified live against Kenya (correctly took the no-op path, confirmed via a second run that `country_initialization_status` was byte-for-byte unchanged). Committed `d0ffefa`, pushed. `pipeline.yml`'s comment corrected to match. This script, not a bare `bruin run`, is what any future cron/CI steady-state schedule must invoke.
4. **Done for Layer B, 2026-07-04.** Add golden-file regression tests validating path A's output against at least one known incident (the Finance Bill 2024 window) before cutting the dashboard over, per `testing-strategy.md` Priority 1. Scoped, per direct investigation, as **Layer B**: a drift check against the already-validated, materialized BigQuery output (not a rerun of the SQL logic against a frozen input snapshot — that "Layer A" design is blocked by two infrastructure gaps, tracked separately as TD-39, since `acled_pressure_regimes.sql` hardcodes its dataset references and `tests.yml` has no GCP credentials wired in). `tests/test_acled_pressure_regimes_golden.py` and its fixtures are committed (`b3ae94d`), confirmed passing against live data and skipping cleanly without the opt-in `RUN_BIGQUERY_TESTS=1` var set. Wiring this into `.github/workflows/tests.yml`'s existing CI job remains a deliberately separate follow-up. **Candidates confirmed as of 2026-07-04, now sorted by what batches 3–4 actually showed, not just what was expected:**
   - **Clear positive detections** (real transition, plausible timing, good golden-file candidates): **the Finance Bill 2024 window is the strongest of all** — MOBILISATION from 2024-05-11 (Bill tabling), fresh CRISIS on 2024-06-22→06-28 (Parliament stormed, SEVERE/SEVERE, HIGH confidence, no lag), sustained through Saba Saba, clean STABLE reversion by 2024-07-13; this should be the primary golden file. Also strong: the Aug 2022 election (fresh CRISIS on election week 2022-08-06, SEVERE/SEVERE/HIGH, reverts by 2022-08-27); the Jan–Feb 2008 post-election violence (ESCALATION/SEVERE/HIGH from 2008-01-19, 4 weeks); the Oct 2017 election rerun (ESCALATION, 2017-10-07→10-28, no confidence decay); the 2015-02-21→04-04 window spanning the Garissa University attack (7 weeks, sustained HIGH/MEDIUM). The 2018-08-04→10-13 window (10 weeks, sustained HIGH/MEDIUM) still needs a named-event check before use.
   - **Genuine true-negative candidates** (no signal, and that's plausibly *correct*, not a gap): the Aug 2010 constitutional referendum (STABLE throughout — a genuinely peaceful vote, unlike 2007/2017) and the Sept 1, 2017 Supreme Court election annulment (STABLE throughout — a procedural/legal event with no street-level violence, plausibly outside what ACLED conflict-event data can see at all). Both are useful as false-positive control cases in the golden-file suite, not failures to fix.
   - **Ambiguous, needs care:** the Sept 21, 2013 Westgate Mall attack falls inside a window that actually started three weeks earlier (2013-08-31) and had already decayed to `INSUFFICIENT_DATA` confidence by the attack date itself — a naive test asserting "ESCALATION during Westgate week" would technically pass, but for the wrong underlying reason. Do not use this as a golden file without first confirming what the pre-existing 08-31 spike actually was.
   - **Unverified, flagged for later:** Claude Code also noted two 2025 windows (2025-06-07/06-14 and 2025-09-13/09-27) that were not part of the checked list and have not been cross-referenced against named events with the same rigor as the above — do not use as golden files until confirmed.
5. **Done, 2026-07-04** (design corrected the same day, based on direct investigation — not a hard cutover). The original framing of this step ("replace path B with path A") assumed a clean drop-in swap. Investigation of the real schemas found this assumption wrong: path B (`fact_country_pressure_daily`) produces a **daily**, **continuous** composite score (three weighted numeric sub-scores + one derived numeric total + a threshold-based category); path A produces a **weekly**, **categorical** regime classification (`primary_regime`, `confidence_level`, transition fields, per-family bands) with no numeric score at all. Forcing path A's output through path B's rolling-baseline/sigmoid arithmetic would be a hidden methodology redesign, not a cutover. Confirmed the full consumer chain first: `fact_country_pressure_daily` feeds not only `political_stress_windows_mart` and `protocol_repression_correlation_mart`, but also `fact_asn_repression_index` → `asn_behavior_profile_mart` → the Streamlit page carrying TD-02's CROSS JOIN bug — a fourth consumer not in the original scoping, now on record.

   **Corrected decision: ship additively, not as a replacement.** Add path A's fields to `fact_country_pressure_daily` via a `LEFT JOIN` to `intelligence.acled_pressure_regimes` (join key: the week containing each `measurement_date`, matched to `week_start_date` — the exact week-anchor convention must be verified against the regime engine's own calendar-spine logic before writing this join, not assumed), under a `regime_*` column prefix, changing none of the existing columns. Do not touch the existing composite-score arithmetic in either reporting mart. Add the new `regime_*` columns to both reporting marts' `SELECT` lists (additive — confirmed safe against `streamlit/core/contracts.py`'s schema guard, which only enforces `required_columns` as a subset). Add a new Streamlit query function and a new dashboard panel rendering path A's regime/confidence/transition data as its own view, rather than repointing the existing numeric KPIs and line chart at fields that don't exist in path A's output (`stress_color()` and `STRESS_LEVELS` are keyed to path B's category names specifically and would silently mis-render — gray fallback — if fed `primary_regime` directly). Add a `primary_regime` color/category map to `core/constants.py`, and add the missing `INSUFFICIENT_DATA` entry to `CONFIDENCE_LEVELS` while there, since path A actually emits it.

   Run both paths side by side in production for an observation period before deciding path B's retirement — that decision is deliberately deferred to a future, separate ADR update, not bundled into this one. Real evidence for why this matters: a direct comparison of both paths over the Finance Bill 2024 window found path B's peak reading for the *entire* window is `ELEVATED` — it never reaches `SEVERE`, not even the week Parliament was stormed and the deadliest day of the unrest occurred. Path A independently and correctly classifies that exact week (2024-06-22) as `CRISIS`, its most severe category, HIGH confidence, clean `ESCALATION` transition. Path B's floor is also propped up throughout the window by `legal_pressure_score` (weight 0.25) staying around 5-6 even in weeks with zero conflict activity — traceable to TD-01's synthetic Lumen data. This is now the strongest concrete argument in this entire ADR for why path A matters, and independently raises the urgency of TD-01 and TD-02 (both already scoped in `implementation-roadmap.md` Step 2, now with direct evidence they affect today's production numbers, not just a theoretical risk).

   **Annotation, 2026-07-05 (ADR-0004 / TD-44 / TD-45):** the "weight 0.25" and "5-6 floor" figures in the paragraph above describe `composite_pressure_score`'s formula as it stood at the time this paragraph was written — accurate then, not rewritten here to preserve the historical record. `legal_pressure_score` is no longer a term in `composite_pressure_score` at all as of ADR-0004's reweighting (Lumen formally benched); the composite is now `conflict_pressure_score * 0.75 + platform_pressure_score * 0.25`. The underlying point this paragraph makes — that path B's floor was inflated by a fabricated, non-evidentiary input — still holds and is in fact the reasoning ADR-0004 acted on; only the specific numbers are now stale. Re-verified directly: the flagship day (2024-06-22) stays `ELEVATED` under the reweighted formula (5.4292 → 5.0989, still short of `SEVERE`), unchanged from the comparison above. Across the full Finance Bill 2024 window (2024-05-11 to 2024-07-13, 64 days), 57 days stay `ELEVATED`, 1 day stays `MODERATE`, and 6 days move `ELEVATED`→`MODERATE` — so "path B's peak reading is `ELEVATED`, never `SEVERE`" remains true, but "the entire window is `ELEVATED`" was already only approximately true before this change (one day, 2024-05-25, was already `MODERATE` under the original weights, since `legal_pressure_score` happened to be `0` that day) and is somewhat less true after it. See ADR-0004's Consequences section and TD-44/TD-45 in `technical-debt-inventory.md` for the full before/after numbers.

   **Implementation, 2026-07-04.** Before writing the join, the week-anchor convention was verified empirically rather than assumed: `intelligence.acled_pressure_regimes.week_start_date` is **Saturday**-anchored, not Monday-anchored as `stg.acled_conflict_events.sql`'s own header comment claims (that comment is stale — 100% of both the staging table's 15,820 Kenya rows and the intelligence table's 1,523 rows fall on a Saturday; `week_start_date` passes through unchanged from staging to intelligence, so the anchor set at ingestion holds end to end). The join uses `DATE_TRUNC(measurement_date, WEEK(SATURDAY)) = week_start_date`. `fact_country_pressure_daily.sql` now carries an additive `regime` CTE and eleven new `regime_*` columns (`regime_primary_regime`, `regime_confidence_level`, `regime_transition_detected`, `regime_transition_type`, `regime_previous_regime`, `regime_protest_band`, `regime_violence_band`, `regime_suppression_band`, `regime_disorder_band`, `regime_weeks_in_current_regime`, `regime_methodology_version`), nullable rather than `COALESCE`d (`NULL` = no classification exists for that week, a different and more honest signal than a fabricated default). Both reporting marts pass these columns through unchanged from `fact_country_pressure_daily`, without touching their own composite-score/baseline/correlation arithmetic. `streamlit/services/marts.py` gained `get_regime_classification()`; `streamlit/core/constants.py` gained a `REGIME_STATES` color map (path A's 7 regimes, kept separate from `STRESS_LEVELS`) and the missing `INSUFFICIENT_DATA` entry in `CONFIDENCE_LEVELS`; `streamlit/core/theme.py` gained `regime_color()`; page 1 (`1_National_Stress_Observatory.py`) gained a new, separate "ACLED Regime Classification" panel — the existing composite-score KPI row, line chart, and `stress_color()` usage are untouched.

   `bruin validate` passes (57/57 assets). The join was verified against real data without materializing `fact_country_pressure_daily` (see below) — an ad-hoc query joining the live table to `intelligence.acled_pressure_regimes` via the corrected key reproduced the known Finance Bill 2024 timeline exactly (`MOBILISATION` 2024-05-11, `CRISIS` 2024-06-22, `STABLE` 2024-07-13, with mid-week dates correctly bucketing back to their Saturday-anchored week), with `conflict_pressure_score`/`composite_pressure_score`/`pressure_level` unchanged.

   **A separate, pre-existing bug surfaced during this verification, deliberately not fixed here (see TD-40):** `fact_country_pressure_daily.sql`'s `acled` CTE groups by `event_date`, which `stg.acled_conflict_events.sql` hardcodes to `NULL` for every row (confirmed empirically: 100% of 15,820 Kenya rows). Since the asset uses `strategy: create+replace`, the *next* materialization of `fact_country_pressure_daily` — for any reason, not just this change — will silently collapse `conflict_events`/`conflict_pressure_score`/`composite_pressure_score` to `0` for every date, because the join key can never match. The live table's current realistic-looking values must predate this staging contract and have not been rebuilt since. This is why the join above was verified via a standalone `SELECT` rather than `bruin run`: materializing today would have masked the additive verification behind an unrelated regression. Fixing it is out of scope for this ADR step (additive-only, no existing-column changes) and is tracked separately as TD-40.

## Consequences

If completed: the dashboard gains a second, methodologically distinct reporting surface (regime classification with confidence/bands) alongside the existing composite-score view, closing the gap between the product's evidentiary claims and what it actually computes — without forcing an unreviewed redesign of the existing scoring arithmetic. The backfill orchestration, precondition-guard, and testing work (steps 1–4) has value independent of this decision — it was already scoped as debt-reduction work regardless of whether path A was ever wired in.

If step 1 reveals the backfill is far from complete, this becomes a materially larger effort than "flip which table a mart reads from" — the roadmap should be read as sequenced accordingly, not compressed into one change. This ADR does not authorize skipping steps 2–4 to wire path A in faster; doing so would reintroduce the exact silent-corruption and unvalidated-confidence risks this decision was meant to close. Step 5's corrected, additive design similarly does not authorize retiring path B's existing composite score without a separate, explicit follow-up decision informed by the observation period.
