# ADR-0002: Wire the sophisticated ACLED regime engine (Path A) into reporting

Status: Accepted
Date: 2026-07-04

## Context

Step 0 of the Implementation Roadmap confirmed, independently, twice — once against a pre-commit snapshot of the repository and once against the live Codespace repo via Claude Code, with identical results — that the dashboard's national pressure score is built entirely from ACLED path B (`stg.acled_conflict_events` → `marts.fact_country_pressure_daily` → `reporting.mart_political_stress_windows`), and does not consume `intelligence.acled_pressure_regimes` (path A) or `intelligence.country_readiness` anywhere. Path A is the more methodologically mature of the two: it assigns a dual `classification_confidence`/`methodology_risk_level` split, applies per-family statistical guardrails, and performs stateful regime classification, none of which path B does. See `methodology-consistency-review.md` Sections 2 and 6, and `technical-debt-inventory.md` TD-03.

Three options existed: wire path A in, formally deprecate it, or defer the decision while hardening path B. Given the product strategy explicitly sells confidence-qualified findings to legal and journalistic buyers, the methodological gap between what is sold and what is actually computed was judged too consequential to leave unresolved or to resolve by discarding the more rigorous path.

## Decision

Wire path A into the reporting layer, replacing path B as the source of the dashboard's national ACLED pressure signal. This is not a single change — it requires, in order:

1. **Done.** Establish the actual current state of `intelligence.acled_pressure_regimes` and `intelligence.country_initialization_status` before anything else — confirmed a partial backfill existed (15/1,523 weeks), not the zero or complete state either extreme would have suggested.
2. **Done, 2026-07-04.** Complete or resume the historical backfill via `Bruin/scripts/historical_initializer/backfill_acled_pressure_regimes.py` for whatever history is missing, since path A's stateful classification requires sequential, committed history to produce meaningful regime labels rather than defaulting every week to `is_first_observation_week`. Kenya is now `COMPLETE`: 1,523/1,523 weeks (1997-01-11 → 2026-03-14), 0 errors across five batches. This produced the first real evidence for this ADR's premise: the classifier cleanly detected the Jan 2008 post-election violence, the Aug 2022 election, and — most importantly for the product — the Finance Bill 2024 protests, with a fresh CRISIS transition landing exactly on the June 22-28 week Parliament was stormed (protest/suppression bands both SEVERE, HIGH confidence, no lag), sustained through Saba Saba, clean reversion after. It also produced two plausible true-negatives (2010 referendum, Sept 2017 court annulment) and confirmed TD-05's confidence-decay question is most likely a data-sparsity artifact of pre-2020 ACLED coverage, not a structural flaw (0 `INSUFFICIENT_DATA` weeks from 2020 onward, across 111 non-STABLE weeks). See the candidate list below for what's usable as a golden file and what still needs care.
3. **Done, 2026-07-04.** Add the runtime precondition guard to `intelligence.acled_pressure_regimes.sql` (TD-04) before this asset becomes load-bearing for a live, sold product — today a precondition violation corrupts state silently, which was tolerable while the asset was unconsumed and is not tolerable once it feeds a client-facing number. The precondition actually lives under the file's "EXECUTION CONTRACT (AUTHORITATIVE)" comment block (not "ARCHITECTURE FREEZE"/BLOCKER-002 as earlier documentation here mistakenly said — corrected). Design: Bruin's native `custom_checks` run *after* asset execution, too late to prevent a corrupting MERGE, so the guard is instead a new upstream Bruin asset (`intelligence.acled_pressure_regimes_precondition_check`) that calls `ERROR()` on violation and is added to the regime asset's `depends:` list — Bruin's DAG then refuses to invoke the regime asset at all on failure, rather than letting it run and corrupt state. Applied and verified against the real repo: `bruin validate` (57/57 assets clean) and `bruin lineage` (new check confirmed upstream of the regime asset). Committed `2080b64`, purely additive — no change to the existing MERGE or materialization logic, and the completed 1,523-week Kenya backfill is untouched.

   **This design work surfaced a more urgent, separate finding — see TD-38, still open.** `Bruin/pipeline.yml`'s documented steady-state default (`target_week=""`) silently violates this same EXECUTION CONTRACT, because the upstream `features.acled_pressure_signals` asset recomputes the entire country history every run (`strategy: create+replace`, no incremental filter) rather than just the newest week. This hasn't fired yet only because no cron/CI schedule currently invokes the pipeline in steady-state mode. The TD-04 guard now applied will catch and loudly block this the moment it would occur — but the underlying contradiction in `pipeline.yml`'s own documented default is a separate, unresolved design problem that must be fixed before any steady-state schedule is ever wired up, not something the guard itself resolves.
4. **Done for Layer B, 2026-07-04.** Add golden-file regression tests validating path A's output against at least one known incident (the Finance Bill 2024 window) before cutting the dashboard over, per `testing-strategy.md` Priority 1. Scoped, per direct investigation, as **Layer B**: a drift check against the already-validated, materialized BigQuery output (not a rerun of the SQL logic against a frozen input snapshot — that "Layer A" design is blocked by two infrastructure gaps, tracked separately as TD-39, since `acled_pressure_regimes.sql` hardcodes its dataset references and `tests.yml` has no GCP credentials wired in). `tests/test_acled_pressure_regimes_golden.py` and its fixtures are committed (`b3ae94d`), confirmed passing against live data and skipping cleanly without the opt-in `RUN_BIGQUERY_TESTS=1` var set. Wiring this into `.github/workflows/tests.yml`'s existing CI job remains a deliberately separate follow-up. **Candidates confirmed as of 2026-07-04, now sorted by what batches 3–4 actually showed, not just what was expected:**
   - **Clear positive detections** (real transition, plausible timing, good golden-file candidates): **the Finance Bill 2024 window is the strongest of all** — MOBILISATION from 2024-05-11 (Bill tabling), fresh CRISIS on 2024-06-22→06-28 (Parliament stormed, SEVERE/SEVERE, HIGH confidence, no lag), sustained through Saba Saba, clean STABLE reversion by 2024-07-13; this should be the primary golden file. Also strong: the Aug 2022 election (fresh CRISIS on election week 2022-08-06, SEVERE/SEVERE/HIGH, reverts by 2022-08-27); the Jan–Feb 2008 post-election violence (ESCALATION/SEVERE/HIGH from 2008-01-19, 4 weeks); the Oct 2017 election rerun (ESCALATION, 2017-10-07→10-28, no confidence decay); the 2015-02-21→04-04 window spanning the Garissa University attack (7 weeks, sustained HIGH/MEDIUM). The 2018-08-04→10-13 window (10 weeks, sustained HIGH/MEDIUM) still needs a named-event check before use.
   - **Genuine true-negative candidates** (no signal, and that's plausibly *correct*, not a gap): the Aug 2010 constitutional referendum (STABLE throughout — a genuinely peaceful vote, unlike 2007/2017) and the Sept 1, 2017 Supreme Court election annulment (STABLE throughout — a procedural/legal event with no street-level violence, plausibly outside what ACLED conflict-event data can see at all). Both are useful as false-positive control cases in the golden-file suite, not failures to fix.
   - **Ambiguous, needs care:** the Sept 21, 2013 Westgate Mall attack falls inside a window that actually started three weeks earlier (2013-08-31) and had already decayed to `INSUFFICIENT_DATA` confidence by the attack date itself — a naive test asserting "ESCALATION during Westgate week" would technically pass, but for the wrong underlying reason. Do not use this as a golden file without first confirming what the pre-existing 08-31 spike actually was.
   - **Unverified, flagged for later:** Claude Code also noted two 2025 windows (2025-06-07/06-14 and 2025-09-13/09-27) that were not part of the checked list and have not been cross-referenced against named events with the same rigor as the above — do not use as golden files until confirmed.
5. Only then, modify the reporting layer (most likely `political_stress_windows_mart.sql` and `protocol_repression_correlation_mart.sql`, and possibly `fact_country_pressure_daily.sql` itself) to source ACLED pressure from `intelligence.acled_pressure_regimes` instead of the current direct read of `stg.acled_conflict_events`.

## Consequences

If completed: the dashboard's national pressure score gains a confidence split, statistical guardrails, and regime persistence it currently lacks, closing the gap between the product's evidentiary claims and what it actually computes. The backfill orchestration and precondition-guard work (steps 1–4) has value independent of this decision — it was already scoped as debt-reduction work regardless of whether path A was ever wired in.

If step 1 reveals the backfill is far from complete, this becomes a materially larger effort than "flip which table a mart reads from" — the roadmap should be read as sequenced accordingly, not compressed into one change. This ADR does not authorize skipping steps 2–4 to wire path A in faster; doing so would reintroduce the exact silent-corruption and unvalidated-confidence risks this decision was meant to close.
