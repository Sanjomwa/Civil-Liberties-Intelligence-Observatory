# ADR-0003: CLIO's Evidence Architecture — Foundation Datasets and Admission Criteria

Status: Accepted (research conclusion; no code changed by this ADR)
Date: 2026-07-05, refined 2026-07-05 (same day, second pass)

## Addendum — evaluation framework refined, Google Transparency resolved

The original evaluation criteria in this ADR treated every candidate as an independent addition. Direct scrutiny of *why* STOP (Access Now's Shutdown Tracker Optimization Project) actually scored Core found that framing incomplete: STOP's value is disproportionate precisely because it attaches to OONI and ACLED's existing relationship, turning "a technical blocking signal and a conflict signal were elevated in the same window" (correlation) into "this was an official act, ordered by a named authority, for a stated reason" (attribution). Its marginal value scales with the number of existing sources it can attach an attribution to — it is not simply a good dataset in isolation.

This produced a real refinement to the criteria: **two distinct, non-substitutable value modes**, not one. **Chain-strengthening** value improves the attributional or confidence quality of conclusions already drawn from existing sources (STOP's and IODA's mode). **Vector-opening** value answers a decision-relevant question existing sources don't ask at all, independent of whether it reinforces anything already present (CPJ's mode — it doesn't triangulate with OONI/ACLED/STOP, it opens "was the press itself directly targeted," a question self-referentially relevant to the journalist persona specifically). Evaluating future candidates only by chain-strengthening would have systematically undervalued CPJ, which is real evidence this refinement is necessary, not cosmetic.

**Google Transparency, studied with the same rigor as STOP and CPJ rather than deferred again** (methodology verified directly, not assumed: government content-removal requests, published in six-month buckets, broken out by country/product/reason category, carrying two figures per bucket — request count and percentage actually complied with). Answered against the framework:

- Actual grain: semi-annual, country/product/reason aggregate. Not incident-dated.
- Unique evidence: the compliance-rate figure is a genuinely new evidentiary category this project had not previously named — not observational, not quite attributional in STOP's specific-order sense, but **consequential**: evidence of whether pressure applied actually succeeded, which nothing else in CLIO's evidence set answers.
- Fit with `Observation → Interpretation → Confidence → Inference`: does not fit at incident grain. A six-month bucket cannot be honestly attributed to a specific multi-week incident window without fabricating precision the source doesn't have — the same category of mistake this project caught and corrected at TD-40, TD-02, and ADR-0002 step 5.
- Triangulation vs. duplication: genuinely new evidence (a legal-outcome signal), not a duplicate of ACLED, OONI, or STOP.

**Conclusion: a fifth tier is needed, not a forced fit into Core/Secondary/Optional/Rejected.** Google Transparency is classified **Contextual/Backdrop**: a legitimate pipeline input, computed and reported strictly at its own six-month grain, feeding a report's surrounding narrative (e.g. "formal legal-removal pressure on this platform rose X% this half") rather than any incident-level score, confidence claim, or composite-score input. It must never be broadcast down to a weekly or incident grain the way ACLED's regime engine honestly broadcasts a real weekly value — Google's real grain is six months, and treating it as finer would repeat, not avoid, this project's now-recurring mistake. Meta, X, and other platform transparency reports should be evaluated against this same Contextual/Backdrop template once actually investigated, not assumed to inherit it without their own check.

## Context

This ADR was reached by deliberately reversing the usual order of reasoning. Rather than starting from "what architecture should CLIO have" or "what datasets exist and could we ingest them," the question was worked backward from the user: `User → Decision → Information required → Evidence needed → Methodology → Architecture`. The two users CLIO should optimize for today — derived, not assumed, from the Project Zero Review's own buyer research (`CLIO_Project_Zero_Review.docx`, Sections 2.1–2.4) — are the investigative journalist and the NGO/human-rights-documentation buyer (the latter including legal teams and human rights commissions, whose decision shape is identical). Election observer missions, academic researchers, and corporate/government buyers are real future users but are explicitly not part of the smallest set CLIO can serve well today, each for a stated reason: observer missions' primary decision is forward-looking and operational, which is the opposite of CLIO's retrospective, non-causal methodology; academics have real but low urgency, since they could largely perform this analysis themselves; corporate/government buyers require sales, legal, and security infrastructure that doesn't exist yet and raise a mission-drift risk the Zero Review already flagged.

The journalist's decision: can I credibly publish or cite that a specific window involved digital-pressure activity alongside a specific event, at a stated confidence level. The NGO/legal decision: should this specific incident be included as documented evidence in a filing or report, at a stated confidence level. Both decisions require the same five things, none of which is a single composite score: a bounded time window and geography; an attributed breakdown of *why* pressure is elevated (conflict-driven, legal-driven, platform-driven); an honest, claim-specific confidence level; visible provenance disclosure for every input; and a citable, stable methodology description.

## Decision

**OONI and ACLED are confirmed as CLIO's foundation datasets** — not because they were already implemented, but because, independently verified against the characteristics below, they are the only two current sources that are both genuine, non-synthetic, independently-provenanced measurements and natural fits for CLIO's `Observation → Interpretation → Confidence → Inference` pattern (independently demonstrated in two separate subsystems this project has now built and validated: OONI's protocol pipeline and ACLED's regime engine). Lumen does not currently meet this bar because its present implementation is synthetic (`scripts/lumen_parquet.py`, `np.random.seed(42)`); the fix already shipped (TD-01) makes that fact honestly visible, it does not resolve the underlying gap. Google Transparency remains an open question pending a methodological audit (not yet performed) of whether its data supports anything beyond a coarse, periodic count.

**Derived admission criteria for any future evidence source**, in the order they matter for the two decisions above, not in generic data-quality order: independent provenance (not reliant on self-reporting by a party that might be the source of the pressure); incident-level resolution (a specific window, not an annual country average); update cadence realistically matched to a days-to-months decision cycle; public methodological transparency and repeatability; external citation quality (already recognized by courts, editors, or fact-checkers independent of CLIO); attribution power — does it say who and why, not just that something happened; corroboration value via a genuinely independent measurement method, not a second table of the same underlying signal; and grain honesty — does adopting it require fabricating precision it doesn't actually have.

**Candidate evidence sources surveyed against these criteria** (real-world status checked directly, not assumed from training data, 2026-07-05):

- **Access Now's Shutdown Tracker Optimization Project (#KeepItOn / STOP)** — Core. Contributes attribution OONI structurally cannot: who ordered a shutdown, why (elections, exams, protests, security), duration, and scope, verified through remote sensing plus news/activist/ISP-statement triangulation — a genuinely independent method from OONI's client-side active probing. Public methodology, event-dated grain, real citation standing (used by UNESCO's World Trends report and Cloudflare Radar's own shutdown-alert partnership).
- **CPJ's database of killed/attacked journalists** — Core (case-level data). A wholly new axis of evidence: person-level, work-relatedness-verified press-freedom violations, self-referentially relevant to the journalist persona specifically. Independently sourced and verified (senior-editor case review per CPJ's own published methodology).
- **CPJ's imprisonment-census data** — Secondary. Historically an annual snapshot; CPJ only began dynamic, continuously-updated tracking in 2025. Needs explicit grain handling on ingestion — the same discipline this project already learned the hard way at TD-40 — before being treated as continuous data.
- **IODA (CAIDA / Georgia Tech)** — Secondary. Detects network outages via BGP routing data, darknet background radiation, and active probing — genuinely independent instrumentation from OONI, but answers substantially the same question ("is the network reachable"), not a new axis of evidence. Valuable as a cross-method confidence booster on OONI's existing findings, not as a first-class Source Intelligence pillar of its own.
- **CIPESA (State of Internet Freedom in Africa / Kenya reporting)** — Rejected as a pipeline source. Real, credible regional policy and legal analysis, but periodic and qualitative, not a structured incident-dated dataset (its one structured tool, Community Fakes, targets AI-generated media, outside CLIO's current scope). Forcing narrative analysis into the Evidence→Interpretation→Confidence pipeline would manufacture structure it doesn't have. Legitimate role: a resource a human report-writer cites by hand in a report's prose, entirely outside the SQL pipeline.
- **Freedom House (Freedom on the Net)** — Optional. Annual, country-level, already correctly characterized in the Project Zero Review as a reference benchmark rather than a competitor. Real citation value as a once-a-year manual directional cross-reference; zero incident-level usefulness; would misrepresent its own precision if pulled into the pipeline at incident grain.
- **Meta/X and other platform transparency reports** — explicitly deferred, not evaluated. The same open methodological question as Google Transparency's, multiplied. Resolve Google's own audit first rather than expanding an unproven pattern.

## Consequences

Two datasets (Access Now/STOP, CPJ's case-level data) are recommended for eventual ingestion as genuine additions to the evidence base — a small number by design, consistent with the instruction that a few excellent sources beat many mediocre ones. Neither has been ingested yet; this ADR records the evaluation and the criteria, not an implementation. Before either is built, each should get the same treatment OONI and ACLED did: its own Evidence → Source Intelligence pipeline, not a shortcut into the existing Fusion layer, and — per this project's now five-times-confirmed pattern — the actual ingestion plan should be investigated against the source's real grain and update behavior before code is written, not assumed from its public documentation alone.

Two items (IODA, Freedom House) are recommended only as manual corroboration/citation aids, not pipeline ingestion — a materially smaller engineering commitment that should not be scoped as if it were a new Source Intelligence pillar.

One candidate (CIPESA) is rejected from the pipeline outright, with its legitimate remaining role named explicitly so it isn't silently lost: manual citation in report narrative.

Google Transparency's status remains genuinely unresolved. This ADR does not answer whether it belongs in the pipeline; it defers that answer to the audit already recommended, and explicitly declines to survey further platform-transparency sources until that audit exists, to avoid multiplying an unresolved question.

This ADR does not authorize building ingestion pipelines for STOP or CPJ immediately. It authorizes treating them as the two next candidates worth a real investigation, in the same disciplined, investigate-before-implementing order every other conclusion in this project has required.

## Addendum — feasibility investigation completed, CPJ's classification materially revised

A full feasibility investigation (2026-07-05) checked both candidates against real fetched data — an actual 317-row STOP CSV export, an actual CPJ API response, and CPJ's own official Data Glossary — rather than against documentation claims alone, per this project's standing discipline. Two findings change this ADR's conclusions.

**CPJ's data is licensed CC BY-NC-ND 4.0 (Attribution-NonCommercial-NoDerivatives), confirmed directly from CPJ's own methodology page.** This was not known when CPJ was classified Core above. NC-ND is a materially serious problem for CLIO specifically: CLIO's business model is selling a derivative, commercial product (fused, confidence-qualified findings), and both "commercial" and "derivative" are exactly what NC-ND restricts. **CPJ's Core classification is downgraded to blocked-pending-legal-resolution.** No further technical investment in CPJ ingestion should happen until CPJ (`press@cpj.org`) has been asked directly whether CLIO's specific intended use — incorporating case-level counts into a commercial, derivative composite finding — is permitted, licensed separately, or not permitted at all. This is a business/legal question, not an engineering one, and this project's own discipline (verify before building) applies here just as much as it does to a schema assumption.

**CPJ also has no official public export or documented API.** The only access path found was an internal REST endpoint (`cpj-datamanager`) discovered by inspecting the site's own network traffic, not published or supported. Independent of the license question, this is a real operational risk: an unpublished internal endpoint could change or be locked down without notice, and building production ingestion against it before even knowing whether CLIO may legally use the data at all would be backwards. Recommendation: raise both questions with CPJ in the same outreach — licensing terms and whether an official bulk-export or supported API exists that CPJ would prefer external consumers use instead.

**STOP's feasibility, by contrast, held up well and sharpened rather than weakened its Core classification.** Real, unauthenticated CSV access confirmed (Google Sheets `/export?format=csv`, 317 live rows fetched), CC BY 4.0 (permissive, no commercial-use conflict), event-level grain confirmed from real data, and a concrete, non-hypothetical piece of triangulation evidence: the Kenya rows in the fetched sample independently cite AS37061, AS36926, and AS33771 by name — the exact three ASNs already tracked as `MAJOR_KENYA_PROVIDER` in CLIO's own `asn_behavior_profile_mart` (confirmed during TD-02's investigation). Real open questions remain before ingestion (whether a single all-years export exists versus per-year tabs with a differing pre-2020 schema, the composite-key dedup rule, how to handle "Unconfirmed" entries, and a courtesy check with Access Now's named methodology contact about scraping cadence) — none of them a blocker, all of them answerable by direct investigation rather than legal negotiation.

A documentation gap was also surfaced during this investigation and should be closed: the "Evidence / Source Intelligence / Fusion / Presentation" framing referenced throughout this project's recent reasoning does not actually exist as a named, written-down model anywhere in the repository — it is an informal mapping onto Bruin's real layers (Evidence = staging + intermediate structural normalization; Source Intelligence = features + intelligence; Fusion = marts; Presentation = reporting + Streamlit), reconstructed independently each time it's needed rather than written once. `docs/02-architecture/README.md` already planned, but never wrote, files that would be its natural home. Worth writing down properly now, given the pattern has been relied on repeatedly (ADR-0002 step 5, this ADR, ADR-0004, and this investigation) — that repetition is itself the two-concrete-cases justification this project requires before treating something as settled enough to formalize.

**Revised recommendation:** proceed with STOP as the next real ingestion candidate — its license, access method, and grain are all confirmed workable, and its Kenya-specific triangulation value is now demonstrated, not just argued. CPJ's technical feasibility is well understood, but implementation should not proceed until the licensing question is resolved directly with CPJ; treat this as a parallel, non-blocking outreach rather than a reason to slow STOP down.
